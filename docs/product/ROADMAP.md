# 信息追踪 Agent 路线图与产品演进（v0 → v3）

迭代节奏：默认 2 周一个 iteration
原则：每期必须"可验收、可回滚、可复盘"，Agent 能力逐步增强而非一口吃成胖子。
核心：Goal-first、少打扰、证据链、Agent 可控（结构化/守门/可回放/可降级）

---

## 版本总览

| 版本 | 定位 | 周期 | 核心目标 |
|------|------|------|----------|
| v0 | 可用 | 2-4 周 | Agent 闭环 + 三层推送跑通 |
| v1 | 好用 | 6-12 周 | 准确性/收敛/模板/健康建议 |
| v2 | 可增长 | 14-30 周 | 多用户、扩源、多渠道、可运营 |
| v3 | 平台化 | 30 周+ | 插件生态、团队能力、工作流输出 |

---

## 目标用户演进

- P0：单人自用
- P1：信息密集型个人（求职/研究/产品/投资/技术跟踪）
- P2：小团队（竞品/市场/政策/安全情报）
- P3：组织级（可选）：更强权限、审计、合规模块（若商业化）

### 长期稳定的核心场景
- 远程岗位追踪：技术栈/公司/地区/薪资等变化
- AI 模型发布：官方 release / 关键 benchmark / 研究突破
- 电车新车/品牌动态：品牌、发布会、预售、量产节点
- 竞品/政策/安全漏洞：高价值变化预警

---

## 北极星指标

**Goal-level Weekly Value Delivered（每Goal每周交付价值）**
代理指标（v0-v1）：每周有效点击/like 数（按 goal_id）

---

## v0（第 1-2 个迭代，2-4 周）—— 可用

### 目标
- 端到端闭环跑通
- Immediate（事件触发）+ Batch（窗口）+ Digest（09:00 CST）
- Push Orchestrator Agent 可回放、可降级

### 关键交付
- 默认源30–40 + RSS 管理
- items 去重、embedding、摘要（候选才做）
- Goal：description + priority_terms（多行）+ windows<=3 + STRICT/SOFT
- Agent：规则分桶 + LLM边界判别（JSON）
- SMTP 发信、redirect 点击统计、like/dislike/block_source

### 退出标准（DoD）
- 连续 48h 稳定运行
- Immediate/Batch/Digest 都能按设计发出，且无重复轰炸
- LLM 不可用时仍能降级运行

### OKR
- O：Immediate/Batch/Digest 三层推送可用、可回溯、可降级
- KR：Agent success_rate >= 95%；48h 稳定运行；Weekly Useful Clicks > 0

---

## v1（第 3-6 个迭代，6-12 周）—— 好用

### 目标
- 更准、更省心、更少噪音
- 降低配置成本，提高"开箱可用"

### 重点能力

#### 1) 模板化与冷启动引导
- 远程岗位/AI发布/电车/竞品：一键生成 description + priority_terms + negative 示例
- Goal 创建向导：让用户 30 秒内完成一个可用 Goal

#### 2) Goal Health（Agent 扩展）
- 自动提示：Goal 太宽（噪音多）、太窄（无结果）、来源质量差
- 给可执行建议：加/删 terms、推荐屏蔽来源、调窗口/阈值建议

#### 3) 收敛与个性化（非黑盒）
- feedback_adjust 更系统：like/dislike 影响排序权重更清晰
- 提供"阈值建议"：系统建议 Immediate 边界区间与规则阈值（仍由用户确认应用）

#### 4) 内容结构增强
- 更强去重：标题相似/内容hash
- 基础实体抽取（品牌/公司/技能）用于 reasons 与过滤

### v1 退出标准
- Immediate CTR 提升、负反馈率下降（以 v0 基线为对照）
- 用户配置步骤减少（模板使用率>一定比例）

### OKR
- O：降低噪音，提高 Immediate 质量
- KR：Immediate CTR +30%；负反馈率 -30%；漏报主观样本下降

---

## v2（第 7-14 个迭代，14-30 周）—— 可增长

### 目标
- 从"单人好用"变成"可复用、可分享、可增长"
- 扩源与多渠道支持

### 重点能力

#### 1) 多用户体系
- 账号体系、数据隔离
- 配额/预算（Goal 数、源数、LLM 调用预算）

#### 2) 扩源插件化（Source Plugin）
- 标准化 Source Adapter 接口：抓取、解析、去重、质量指标
- 官方渠道优先（GitHub Releases、ArXiv、官方博客、RSSHub 等可选）

#### 3) 多渠道推送（可选）
- Telegram/Slack（比邮件更及时）
- 仍保留 Email 作为 "可靠兜底"

#### 4) 运营与增长
- 新手引导漏斗：注册→创建Goal→首封点击→首个like
- 关键行为埋点与看板

### v2 退出标准
- 新用户创建 Goal 成功率与留存达到目标阈值
- 成本/用户可控（预算机制生效）

### OKR
- O：支持多用户、模板化 onboarding、扩源与多渠道
- KR：新用户首日 Goal 创建率 > 60%；周留存 > 30%（视实际调整）

---

## v3（可选，30 周+）—— 平台化 / 生态

### 目标
- 插件生态 + 团队能力 + 工作流输出

### 重点能力
- 插件 marketplace（源/渠道/渲染器）
- 团队权限与审计（如需要商业化）
- Workflow 输出：周报、变化对比、趋势（严格证据链、低风险）

### v3 退出标准
- 插件接入成本显著降低（文档/模板/测试工具完善）
- 团队场景可用（权限/审计/稳定性）

### OKR
- O：插件化源/渠道/渲染器，形成可扩展系统
- KR：第三方源接入成本下降；团队权限/审计可用（若需要）

---

## 能力地图（Capability Map）

### 信息获取层
- Source 管理：默认源、RSS、（v2）插件源
- 抓取质量：退避/熔断、去重、解析监控
- 内容处理：摘要、embedding、（v1+）语言检测/分类/实体抽取

### Goal 模型层
- Goal 描述与约束：must/nice/negative（v0 可空）
- 超高价值标准：priority_terms + mode（STRICT/SOFT）
- 范围：all / selected sources /（v1+）groups /（v2+）topic filters

### 匹配与排序层
- v0：可解释 match_score + reasons
- v1：基于反馈动态调权、阈值建议
- v2：个性化 rerank（按Goal/用户习惯）

### 推送与渠道层
- v0：Immediate/Batch/Digest 三层推送（Email）
- v1：更细策略（静默、频率自适应、热点合并）
- v2：多渠道（Telegram/Slack/Mobile push 可选）

### Agent 层（贯穿演进）
- v0：Push Orchestrator Agent（推送调度+边界判别）
- v1：Goal Health Agent（Goal 太宽/太窄/噪音源建议）
- v2：Source Quality Agent（源质量评分、淘汰/新增建议）
- v3：Workflow Agent（低风险工作流输出：周报/对比/趋势）

---

## 产品原则（长期不变）

1) 少打扰：即时推送只给超高价值
2) 证据优先：判别/总结必须可追溯到 items
3) Agent 可控：结构化输入输出、硬规则守门、失败可降级
4) 成本可控：LLM 分层调用、边界才用、支持预算
5) 安全红线：不自动执行高风险动作；防 prompt injection（隔离外部内容）

---

## 指标体系

- Activation：创建Goal → 首封邮件点击 → 首次 like
- Retention：周活跃 Goal、连续点击周数
- Quality：Immediate CTR、负反馈率、漏报样本
- Efficiency：LLM 调用/成本、每Goal日均邮件封数
- Growth（v2+）：注册→创建Goal→首周留存、模板使用率

---

## 风险与治理

- 误报/打扰：三层推送 + 守门 + 反馈收敛
- 内容安全：输入隔离、引用限制、结构化输出
- 成本：分层调用、边界才用、预算与熔断
- 扩源噪音：源质量评分、自动建议淘汰/新增（v1+）
- 多用户合规：权限/数据隔离/审计（v2+）

---

## 复盘与更新机制

每个迭代复盘必须回答：
1) 本期最关键假设是否成立？证据是什么？
2) 下一期最要解决的"一个问题"是什么？
3) 需要调整路线图吗？（如有，更新本文件）

每次迭代产出：
- Release notes：变更点/风险/回滚
- 指标看板：本迭代 3 个关键指标
- 复盘：假设是否成立、下一步调参/改设计
