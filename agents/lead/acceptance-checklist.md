# 验收清单

> Lead Agent 在交付前使用此清单验收工作成果。

---

## ✅ 通用验收项

每次交付前必须检查：

### 1. 功能完整性

```
□ 功能是否按需求实现？
□ 边界情况是否处理？
□ 错误情况是否有合理的反馈？
```

### 2. 代码质量

```
□ 代码是否符合项目规范？
□ 命名是否清晰、一致？
□ 是否有明显的代码重复？
□ 注释是否必要且有意义？
```

### 3. 类型安全

```
□ TypeScript/Python 类型是否完整？
□ 是否避免了 any/Any 的滥用？
□ 接口定义是否准确？
```

### 4. 测试

```
□ 关键逻辑是否有单元测试？
□ 现有测试是否仍然通过？
□ 是否需要补充集成测试？
```

### 5. 文档

```
□ 是否需要更新 API 文档？
□ 是否需要更新 README？
□ 复杂逻辑是否有注释说明？
```

---

## 🎨 前端特定验收

```
□ 组件是否响应式？（mobile / tablet / desktop）
□ 加载状态是否处理？
□ 错误状态是否处理？
□ 空状态是否处理？
□ 是否符合 UI 设计规范？
□ 是否有可访问性问题？（键盘导航、屏幕阅读器）
```

---

## ⚙️ 后端特定验收

```
□ API 接口是否符合 RESTful 规范？
□ 错误响应是否结构化？
□ 是否有必要的日志？
□ 是否考虑了并发安全？
□ 数据库查询是否高效？（N+1 问题）
□ 是否需要添加索引？
```

---

## 🗄️ 数据库变更验收

```
□ 迁移脚本是否可逆？
□ 是否考虑了数据兼容性？
□ 索引是否合理？
□ 是否需要数据迁移？
```

---

## 📝 交付报告模板

验收通过后，使用以下模板交付：

```markdown
## ✅ 任务完成

**需求**：[简述需求]

**完成内容**：
- [x] [具体完成项 1]
- [x] [具体完成项 2]

**修改文件**：
- `path/to/file1.ts` - [修改说明]
- `path/to/file2.py` - [修改说明]

**验收状态**：
- [x] 功能完整性
- [x] 代码质量
- [x] 类型安全
- [ ] 测试（如适用）
- [ ] 文档更新（如需要）

**注意事项**：
[如果有需要用户注意的事项]
```

---

## ⚠️ 验收失败处理

如果验收发现问题：

1. **记录问题**：明确说明什么不符合要求
2. **说明原因**：为什么这是个问题
3. **给出建议**：如何修复
4. **重新执行**：修复后再次验收
